DOM事件流

 |  2024-02-18

DOM2 Events 规范规定事件流分为3个阶段：事件捕获、到达目标、事件冒泡。虽然 DOM2 Events 规范规定的事件流是从 document 开始，但实际上，所有浏览器都是从 window 对象就开始了。

EventTarget

在理解事件流之前，需要先理解 EventTarget。为什么要先理解 EventTarget？因为注册事件监听器（处理程序），大多通过调用 addEventListener()方法或设置 onevent属性这两种方式。而 addEventListener()其实就是 EventTarget.prototype上的方法。主要包括 addEventListener()、 removeEventListener()、 dispatchEvent()。

为什么 DOM元素、 window对象等都可以使用 addEventListener()注册事件监听器，是因为他们都继承了 EventTarget，包括 XMLHttpRequest、 AudioNode 和 AudioContext等等。

事件流

文章开头提到，事件流分为3个阶段：事件捕获、到达目标、事件冒泡。在使用 addEventListener()注册事件监听器的时候，可以通过参数配置，添加到捕获或冒泡阶段，默认注册到冒泡阶段。

但对于 &#x5230;&#x8FBE;&#x76EE;&#x6807;阶段有点特殊。虽然 DOM2 Events 规范明确捕获阶段不命中事件目标，但现在的浏览器都会在捕获阶段在事件目标上触发事件，因此事件目标在 &#x4E8B;&#x4EF6;&#x6355;&#x83B7;、 &#x4E8B;&#x4EF6;&#x5192;&#x6CE1;阶段都会处理事件。就像 document.body clicked &#x4E8B;&#x4EF6;&#x6355;&#x83B7;&#x9636;&#x6BB5;所在的事件可以理解为在 &#x4E8B;&#x4EF6;&#x6355;&#x83B7;阶段处理，而 document.body clicked &#x4E8B;&#x4EF6;&#x5192;&#x6CE1;&#x9636;&#x6BB5;所在的事件在 &#x4E8B;&#x4EF6;&#x5192;&#x6CE1;阶段处理。

Event

在 DOM 中发生事件时，所有相关信息都会被收集并存储在一个名为 event的对象中， event也是传给事件监听器的唯一参数。在通过 dispatchEvent方法触发事件监听时，就用到 Event类生成一个 event对象。不同的事件生成的事件对象也许会包含不同的属性和方法，但他们无疑都继承了 Event。

可以看到， focus对应的事件对象就是 Event类的实例，而 click对应的事件对象是 PointerEvent类的实例，但 PointerEvent也是继承了 Event。

所有的事件对象都包含一些公共属性和方法，例如：

  • type: 事件类型
  • currentTarget: 注册该事件监听器的元素
  • target: 事件目标
  • preventDefault(): 阻止默认行为
  • stopPropagation(): 阻止事件流传播，取消后续的事件捕获或冒泡
  • stopImmediatePropagation(): 阻止事件流传播，并阻止调用任何后续的事件监听 &#xFF08;DOM3 Events &#x4E2D;&#x65B0;&#x589E;&#xFF09;
  • eventPhase: 调用当前事件监听所处的阶段： 1代表捕获阶段， 2代表到达目标， 3代表冒泡阶段。

点击 body时，使用普通函数注册的事件监听，在执行期间 this等于 currentTarget，指向注册当前事件监听的元素。而 target永远指向事件目标 document.body。只有当 &#x5230;&#x8FBE;&#x76EE;&#x6807;阶段，也就是 eventPhase === 2时， this、 currentTarget、 target这三者才是完全相等的，都指向注册当前事件监听器的元素。 Vue事件修饰符中的 self对应的就是这种情况。

event.stopPropagation()方法用于阻止事件流传播，事件流不会传播到下一个元素，更不会传播到下一个阶段。

点击 body时，打印 document clicked &#x6355;&#x83B7;&#x9636;&#x6BB5;、 document clicked &#x6355;&#x83B7;&#x9636;&#x6BB5;-1，因为在 document捕获阶段就已经阻止了事件流传播，不会传播到下一个元素，因此不会触发 document.body的事件监听。

点击 body时，只会打印 document.body clicked &#x6355;&#x83B7;&#x9636;&#x6BB5;，因为在 document.body捕获阶段就已经阻止了事件流传播，不会传播到 &#x5192;&#x6CE1;&#x9636;&#x6BB5;，因此不会触发 document.body冒泡阶段的事件监听。

event.stopImmediatePropagation()方法同样可以阻止事件流传播，但与 event.stopImmediatePropagation()不同的是， event.stopImmediatePropagation()还会阻止调用任何后续的事件监听。什么意思呢？还使用 event.stopPropagation()的第一个例子，将其中的 event.stopPropagation()替换成 event.stopImmediatePropagation()。

此时点击 body，只会打印 document clicked &#x6355;&#x83B7;&#x9636;&#x6BB5;，而在 event.stopPropagation()的例子中会多打印一个 document clicked &#x6355;&#x83B7;&#x9636;&#x6BB5;-1。就是因为 event.stopImmediatePropagation()会阻止调用任何后续的事件监听，因此 document clicked &#x6355;&#x83B7;&#x9636;&#x6BB5;-1对应的事件监听并不会执行，而 event.stopPropagation()则会将当前 &#x5143;&#x7D20;上的当前 &#x4E8B;&#x4EF6;&#x6D41;&#x9636;&#x6BB5;的所有事件监听全部执行。

需要注意的是: event对象只在事件监听器执行期间存在，一旦执行完毕，就会被销毁；调用 stopPropagation()、 stopImmediatePropagation()等方法时，必须通事件对象 event，或者将内部的 this绑定为 event。
